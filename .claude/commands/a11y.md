# /a11y — Auditoria de accesibilidad (WCAG)

Analiza el frontend del proyecto para detectar problemas de accesibilidad segun WCAG 2.1.

## Uso
```
/a11y                    → Auditoria completa del proyecto
/a11y page /ruta         → Auditar una pagina/ruta especifica
/a11y component Button   → Auditar un componente especifico
/a11y fix                → Corregir problemas encontrados automaticamente
/a11y report             → Generar reporte detallado
```

## Instrucciones

### 1. Analisis estatico del codigo

Buscar en TODOS los archivos de componentes (.tsx, .jsx, .vue, .svelte, .html):

**Imagenes sin alt:**
```
<img> sin atributo alt
<Image> (Next.js) sin alt
Iconos decorativos sin aria-hidden="true"
```

**Formularios sin labels:**
```
<input> sin <label> asociado o aria-label
<select> sin label
<textarea> sin label
Botones de formulario sin texto visible o aria-label
```

**Estructura semantica:**
```
Multiples <h1> en una pagina
Saltos en niveles de heading (h1 → h3 sin h2)
<div> o <span> usados como botones sin role="button"
Falta de landmarks (<main>, <nav>, <header>, <footer>)
Listas que no usan <ul>/<ol>/<li>
```

**Contraste y colores:**
```
Texto que depende solo del color para comunicar significado
Clases de Tailwind con contraste insuficiente (text-gray-300 en bg-white)
Focus rings removidos (outline-none sin alternativa)
```

**Interactividad:**
```
onClick en elementos no interactivos (<div onClick>)
Elementos interactivos sin tabIndex
Modales sin focus trap
Dropdowns sin keyboard navigation
Falta de aria-expanded, aria-haspopup, aria-controls
```

**Navegacion por teclado:**
```
tabIndex > 0 (anti-patron)
Elementos con tabIndex negativo que deberian ser accesibles
Falta de skip-to-content link
```

### 2. Analisis con herramientas (si disponibles)

Si el proyecto tiene dependencias de testing:

**Con Playwright:**
```bash
# Correr axe-core via Playwright
npx playwright test --grep accessibility
```

**Con Jest + Testing Library:**
```bash
# Buscar tests de accesibilidad existentes
# jest-axe, @testing-library/jest-dom
```

**Con Lighthouse (si Chrome disponible):**
```bash
npx lighthouse http://localhost:3000 --only-categories=accessibility --output=json
```

### 3. Clasificacion de hallazgos

Clasificar cada hallazgo por nivel WCAG:

| Nivel | Significado | Prioridad |
|-------|-------------|-----------|
| **A** | Minimo requerido — barreras criticas | CRITICO |
| **AA** | Estandar recomendado (objetivo comun) | ALTO |
| **AAA** | Nivel ideal (no siempre alcanzable) | MEDIO |

### 4. Reporte

Generar tabla de hallazgos:

| # | Problema | Archivo | Linea | WCAG | Nivel | Fix |
|---|----------|---------|-------|------|-------|-----|
| 1 | img sin alt | Hero.tsx | 15 | 1.1.1 | A | Agregar alt descriptivo |
| 2 | form sin label | Login.tsx | 23 | 1.3.1 | A | Agregar label o aria-label |
| 3 | heading skip h1→h3 | About.tsx | 8 | 1.3.1 | A | Usar h2 |
| 4 | low contrast text | Card.tsx | 42 | 1.4.3 | AA | Cambiar text-gray-400 → text-gray-600 |
| 5 | div as button | Menu.tsx | 31 | 2.1.1 | A | Usar <button> o agregar role+tabIndex |

**Resumen:**
- Nivel A (critico): X hallazgos
- Nivel AA (alto): X hallazgos
- Nivel AAA (medio): X hallazgos
- Score estimado: XX/100

### 5. Comando `fix`

Si se solicita fix automatico:
1. Mostrar lista de hallazgos con fix automatico disponible
2. Pedir confirmacion antes de aplicar
3. Aplicar fixes uno por uno:
   - Agregar `alt=""` a imagenes decorativas, alt descriptivo a informativas
   - Agregar `aria-label` a inputs sin label
   - Cambiar `<div onClick>` a `<button>`
   - Agregar `aria-hidden="true"` a iconos decorativos
   - Agregar focus ring visible donde se removio
4. Correr tests despues de cada fix para verificar que no rompe nada

### Criterios WCAG clave a verificar

**Perceptible:**
- [ ] 1.1.1 — Contenido no textual tiene alternativa (alt, aria-label)
- [ ] 1.3.1 — Info y relaciones via estructura semantica (headings, landmarks, labels)
- [ ] 1.4.1 — Color no es el unico medio visual
- [ ] 1.4.3 — Contraste minimo 4.5:1 (texto normal), 3:1 (texto grande)
- [ ] 1.4.11 — Contraste de componentes UI 3:1

**Operable:**
- [ ] 2.1.1 — Todo es operable por teclado
- [ ] 2.4.1 — Mecanismo para saltar bloques (skip link)
- [ ] 2.4.2 — Paginas tienen titulos descriptivos
- [ ] 2.4.3 — Orden de focus es logico
- [ ] 2.4.6 — Headings y labels son descriptivos
- [ ] 2.4.7 — Focus es visible

**Comprensible:**
- [ ] 3.1.1 — Idioma de la pagina definido (<html lang>)
- [ ] 3.2.1 — Al hacer focus no hay cambios inesperados
- [ ] 3.3.1 — Errores de input se identifican claramente
- [ ] 3.3.2 — Labels o instrucciones para inputs

**Robusto:**
- [ ] 4.1.1 — Markup es valido (no IDs duplicados)
- [ ] 4.1.2 — Componentes tienen nombre, rol y valor programatico

## Reglas
- No sugerir cambios que rompan el diseno visual — accesibilidad Y estetica
- Priorizar Nivel A (barreras criticas) antes que AA o AAA
- Si un componente tiene tests, verificar que los fixes no los rompen
- Considerar responsive — verificar tanto desktop como mobile
- Los fixes automaticos solo aplican a problemas con solucion clara y segura
